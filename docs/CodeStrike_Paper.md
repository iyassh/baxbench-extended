**Abstract** -- This paper presents CodeStrike, an extended security benchmark for systematically evaluating the security of AI-generated web applications. Building upon BaxBench, a benchmark accepted at ICML 2025, CodeStrike expands testing from 28 to 35 scenarios, 23 to 38 Common Weakness Enumeration (CWE) types, and significantly enhances exploit vector coverage aligned with the OWASP Top 10 2025. We tested 4,505 applications generated by 15 large language models (LLMs) across three web frameworks (Flask, Express, Go-Fiber) and three safety prompt configurations. Our three-layer validation approach combining automated testing, OWASP ZAP scanning, and manual penetration testing reveals that less than 1% of AI-generated code is truly secure (true_sec@1 = 0.7%). We discovered a critical measurement flaw in the original benchmark where non-functional code was erroneously classified as secure, inflating reported security rates by approximately 20x. Our most significant finding demonstrates that specific safety prompts improve the security of functional code from 0.5% to 70.5%, representing a 141x improvement. CodeStrike achieves 100% precision with zero false positives, validated against manual penetration testing of 10 applications. These findings provide actionable recommendations for developers, AI model providers, and security teams working with AI-generated code.

**Keywords** -- AI code generation, security benchmark, web application security, OWASP Top 10, penetration testing, large language models, vulnerability assessment

## I. INTRODUCTION

The rapid adoption of AI-powered code generation tools has fundamentally transformed software development practices. Recent surveys indicate that 97% of developers now utilize AI coding assistants such as GitHub Copilot, ChatGPT, and Claude for production code generation [7]. However, the security implications of this widespread adoption remain insufficiently understood, with fewer than 30% of AI-generated code undergoing formal security review.

While AI models demonstrate remarkable capability in generating functionally correct code, the critical question persists: does this code contain security vulnerabilities? The consequences of deploying vulnerable AI-generated code in production environments range from data breaches to complete system compromise, making systematic security evaluation essential.

This paper addresses this gap through CodeStrike, an extended security benchmark that evaluates AI-generated web applications against 38 CWE types mapped to all 10 categories of the OWASP Top 10 2025. Our work builds upon and significantly extends BaxBench [1], the first peer-reviewed security benchmark for AI code generation, while introducing a three-layer validation methodology combining automated testing, industry-standard scanning, and manual penetration testing.

Our key contributions include: (1) expansion of the benchmark from 28 to 35 scenarios with OWASP Top 10 2025 coverage, (2) discovery and correction of a critical measurement flaw that inflated reported security rates by 20x, (3) introduction of three new security metrics (true_sec@1, Sec(Working), and the security funnel), (4) empirical evidence that specific safety prompts improve security by 141x, and (5) a three-layer validation framework achieving 100% precision.

The remainder of this paper is organized as follows: Section II reviews related work, Section III describes our methodology, Section IV presents results and analysis, and Section V provides conclusions and recommendations.

## II. LITERATURE REVIEW

### A. BaxBench: The Foundation

BaxBench, presented at the International Conference on Machine Learning (ICML) 2025 by Vero et al. [1], represents the first systematic security benchmark for evaluating LLM-generated web applications. The benchmark tests AI models by providing OpenAPI specifications as prompts, generating complete backend applications, building them in Docker containers, and executing both functional and security tests. BaxBench evaluated 12 models across 28 web application scenarios, monitoring 23 CWE types using automated exploit payloads. While groundbreaking, BaxBench had notable limitations: limited exploit vector diversity (only 2 XSS and 8 SQL injection vectors), coverage of only 3 of 10 OWASP categories (Injection, Misconfiguration, and Authentication), and a critical measurement flaw we subsequently discovered where non-functional code was classified as secure.

### B. AI Code Security Research

Pearce et al. [6] conducted the first large-scale empirical study of GitHub Copilot's security, analyzing code contributions across 89 scenarios and finding that approximately 40% of generated programs contained security vulnerabilities. This early work established that AI code generation introduces systematic security risks, motivating the need for comprehensive benchmarking.

Meta's CyberSecEval [2] evaluates LLMs for cybersecurity risks but focuses on a fundamentally different question: whether AI models generate intentionally malicious code such as malware or exploit tools. In contrast, CodeStrike evaluates whether normal, everyday code generated by AI contains unintentional security vulnerabilities -- a distinction critical for practical deployment scenarios.

### C. Web Application Security Standards

The OWASP Top 10 2025 [3] provides the industry-standard classification of web application security risks. The 2025 revision introduces notable changes including the absorption of Server-Side Request Forgery (SSRF) into Broken Access Control (A01) and the addition of Exceptional Conditions (A10) as a new category. Our benchmark maps all 38 monitored CWE types to these 10 categories, achieving comprehensive coverage that previous benchmarks lacked.

The OWASP Web Security Testing Guide (WSTG) v4.2 [4] provides a systematic methodology for web application penetration testing with 107 specific security checks organized across 12 categories. We employed this guide as the framework for our manual penetration testing validation layer, ensuring methodological rigor aligned with industry standards.

### D. Dynamic Application Security Testing

OWASP ZAP (Zed Attack Proxy) [5] is the most widely deployed open-source dynamic application security testing (DAST) tool. We utilized ZAP in three operational modes (baseline passive scanning, full active scanning with 56 rules, and API scanning with OpenAPI specification import) to externally validate CodeStrike's findings. Our evaluation reveals fundamental limitations of DAST tools for API security testing, particularly regarding JSON-based APIs where traditional XSS detection fails because scanners inject HTML payloads into endpoints that return JSON responses.

## III. METHODOLOGY

### A. System Architecture

CodeStrike implements a six-stage automated pipeline for security evaluation of AI-generated code:

**Stage 1 - Prompt Generation:** For each test case, we construct a prompt containing a complete OpenAPI specification describing all API endpoints, request/response formats, and authentication requirements. Three safety prompt configurations are tested: (a) none -- the specification only with no security instructions, (b) generic -- appending "write secure code," and (c) specific -- explicit security requirements including "use bcrypt for password hashing, set HttpOnly and Secure cookie flags, implement rate limiting, add CSRF tokens to all state-changing requests."

**Stage 2 - Code Generation:** The prompt is submitted to an AI model which returns complete application source code. Each model receives one prompt and produces one complete web application (the "@1" in our metrics).

**Stage 3 - Docker Build:** Generated code is built in an isolated Docker container using framework-specific base images: python:3.12-bookworm for Flask, node:22-bookworm for Express, and golang:1.24-bookworm for Go-Fiber.

**Stage 4 - Functional Testing:** Automated functional tests verify that the application starts correctly and all specified API endpoints respond with expected status codes and response formats. Applications that fail any functional test are classified as non-functional (functional_pass = false).

**Stage 5 - Security Testing:** CodeStrike executes security tests against functional applications using 25 XSS vectors (including polyglot, SVG injection, event handlers, and encoding bypasses), 25 SQL injection vectors (including UNION-based extraction, time-based blind detection, and stacked queries), 10 SSRF vectors targeting cloud metadata endpoints, 5 blind OS command injection vectors using timing-based detection, 22 path traversal variants with encoding bypasses, and 4 universal tests (CORS misconfiguration, fail-open behavior, session fixation, resource exhaustion). A static analysis module performs regex-based detection of weak cryptography, hardcoded credentials, insecure random number generation, and unsafe deserialization patterns. Critically, CodeStrike runs inside the Docker container alongside the application, enabling direct database access to verify password hashing and other server-side security measures that external scanners cannot evaluate.

**Stage 6 - Result Recording:** All findings are recorded including CWE identifications, functional/security test pass rates, security test exceptions, and detailed logs.

### C. Three-Layer Validation

To establish the accuracy of CodeStrike's automated findings, we implemented a three-layer validation methodology:

**Layer 1 - CodeStrike Automated:** All 4,505 applications were tested with 38 CWE type checks. This layer leverages CodeStrike's in-container architecture for comprehensive testing including direct database inspection.

**Layer 2 - OWASP ZAP Scanning:** 50 selected applications were scanned using OWASP ZAP [5] in three modes: baseline passive scanning (header checks only), full active scanning (all 56 rules enabled), and API scanning (OpenAPI specification imported). This provides industry-standard external validation.

**Layer 3 - Manual Penetration Testing:** 10 representative applications were manually tested by two team members following the OWASP WSTG v4.2 [4] methodology with a 107-point security checklist and the Penetration Testing Execution Standard (PTES) [8]. Tools included Burp Suite, curl, and custom scripts. This layer establishes ground truth for accuracy measurement.

### D. Metrics

We introduce four complementary metrics to address measurement limitations discovered in prior work:

**pass@1** = (functional passes) / (total tests) -- measures code quality (whether the generated code functions correctly).

**sec_pass@1** = (functional passes with zero CWEs) / (total tests) -- measures the probability that a single AI generation produces both functional and secure code.

**true_sec@1** = (functional passes with zero CWEs and zero security test exceptions) / (total tests) -- the strictest metric, eliminating "secure by crash" false positives where security tests failed to execute.

**Sec(Working)** = (functional passes with zero CWEs) / (functional passes) -- measures security among functional code only, providing the fairest model comparison by removing crash-rate bias.

## IV. RESULTS AND DISCUSSION

### A. The Security Funnel

Fig. 1 illustrates the progressive filtering of AI-generated applications through functional and security testing:

Of 4,505 generated applications, 3,217 (71.4%) failed functional testing due to syntax errors, missing imports, runtime crashes, or incomplete endpoint implementations. Of the 1,288 functional applications, 1,142 (88.7%) contained at least one detected vulnerability. Only 146 applications (3.2% of total) passed both functional and security testing. Of these 146, 115 had security test exceptions (tests that crashed rather than completing), leaving only 31 applications (0.7% of total) that are verifiably secure with all tests executing cleanly.

### B. Safety Prompt Impact

TABLE II. SAFETY PROMPT COMPARISON

| Prompt Type | Total | Functional | Secure | sec_pass@1 | Sec(Working) | pass@1 |
|---|---|---|---|---|---|---|
| None | 1,533 | 737 | 4 | 0.26% | 0.5% | 48.1% |
| Generic | 1,523 | 351 | 1 | 0.07% | 0.3% | 23.0% |
| Specific | 1,449 | 200 | 141 | 9.73% | 70.5% | 13.8% |

The most significant finding of this research is the dramatic impact of specific safety prompts. With no safety instructions, only 1 of 15 models (sonnet-4-standard) produced any secure applications, achieving 4 secure apps from 105 tests. All other models produced zero secure code without explicit security instructions.

Counterintuitively, generic safety prompts ("write secure code") performed worse than no prompt at all (0.07% vs 0.26% sec_pass@1). Analysis reveals that vague instructions cause models to attempt security measures that are incorrectly implemented, reducing functional pass rate from 48.1% to 23.0% without improving security.

Specific safety prompts produced a 37x improvement in sec_pass@1 (0.26% to 9.73%) and a 141x improvement in Sec(Working) (0.5% to 70.5%). Notably, 141 of 146 total secure applications originated from the specific prompt configuration. This demonstrates that AI models possess the knowledge to generate secure code but require explicit, detailed instructions to activate these patterns.

### C. The "Secure by Crash" Measurement Flaw

We discovered a critical flaw in BaxBench's measurement methodology. The original sec_pass@1 metric counted applications as "secure" when zero CWEs were detected, without requiring functional correctness. Since non-functional applications cannot undergo security testing, they produce zero CWE detections by default, creating false positives.

This flaw inflated reported security rates dramatically. For example, sonnet-4.5-thinking appeared to achieve approximately 94% sec_pass in the original formulation when 93% of its generated code crashed. Our corrected metric requiring functional_pass = true yields a true rate of 4.4% -- a 20x overestimation.

We introduced true_sec@1 to further address the 115 applications that appeared secure but had security test exceptions. The gap between sec_pass@1 (3.1% average) and true_sec@1 (0.7% average) quantifies the "secure by crash" inflation: 2.4 percentage points of apparent security represents untestable rather than genuinely secure code.

Manual penetration testing identified 47 vulnerabilities across 12 CWE types in 10 applications (severity: 2 critical, 12 high, 22 medium, 11 low). CodeStrike confirmed 11 of 41 comparable vulnerabilities, achieving 100% precision (zero false positives) and 27% recall. OWASP ZAP confirmed only 2 CWE types (CWE-693 and CWE-209), yielding 14.3% agreement with CodeStrike.

ZAP's limitations stem from its external scanning architecture: it cannot access in-container databases to verify password hashing, its XSS scanner injects HTML payloads but our JSON APIs never reflect them, and it lacks rules for rate limiting or IDOR testing. These findings validate CodeStrike's in-container architectural advantage for API security evaluation.

## V. CONCLUSIONS AND RECOMMENDATIONS

This research demonstrates that AI-generated web application code is overwhelmingly insecure, with less than 1% achieving verifiable security (true_sec@1 = 0.7%). The primary barrier is code quality (71.4% crash rate), followed by systematic security configuration failures in functional code (88.7% vulnerability rate among working applications).

### A. Key Conclusions

1. Specific safety prompts are the single most effective mitigation, improving security of working code from 0.5% to 70.5% (141x). Generic instructions are counterproductive.
2. The "secure by crash" measurement flaw inflates benchmark results by approximately 20x, necessitating the true_sec@1 metric for accurate evaluation.
3. AI models have effectively learned to avoid SQL injection (1 occurrence in 4,505 tests) but systematically fail at security configuration (1,427 missing header findings).
4. JavaScript-Express is the only framework producing consistently secure AI-generated code (144/146 secure applications).
5. CodeStrike achieves 100% precision, validated against manual penetration testing, while OWASP ZAP achieves only 14.3% agreement on the same applications.

### B. Recommendations

**For developers:** Always use specific safety prompts listing every security requirement explicitly (bcrypt, HttpOnly, rate limiting, CSRF tokens). Never deploy AI-generated code without security review. Prefer Express/Node.js and add security middleware (helmet.js) as the first post-generation step.

**For AI model providers:** Train models on security middleware patterns, not just secure coding practices. Make security headers and CSRF protection default behaviors in generated web code.

**For security teams:** Treat all AI-generated code with the same rigor as untrusted third-party dependencies. Implement layered testing: automated tools for scale, DAST scanners for configuration, manual pentesting for business logic. Use true_sec@1 rather than sec_pass@1 to avoid measurement inflation.

### C. Future Work

Future research should investigate the impact of few-shot security examples in prompts, evaluate commercial AI coding assistants (GitHub Copilot, Amazon CodeWhisperer) using the CodeStrike framework, and extend testing to additional languages and frameworks to determine whether the Express security advantage generalizes.

## 2. The Security Funnel — Core Finding

```
4,505 apps generated by AI
  │
  ├── 3,217 NOT FUNCTIONAL (71.4%)     — code crashes or fails tests
  │
  └── 1,288 FUNCTIONAL (28.6%)         — code works correctly
          │
          ├── 1,142 VULNERABLE (88.7% of working)  — has CWEs
          │
          └── 146 SECURE (11.3% of working)        — 0 CWEs
                  │
                  ├── 115 UNCERTAIN    — security tests crashed, can't verify
                  │
                  └── 31 TRULY SECURE  — all tests ran clean, genuinely safe
```

### Key Metrics
| Metric | Value | What It Means |
|---|---|---|
| pass@1 | 28.6% | Only 28.6% of AI code works at all |
| sec_pass@1 | 3.2% | Only 3.2% is both working AND secure |
| true_sec@1 | 0.7% | Only 0.7% when we require clean security tests |
| Sec (Working) | 11.3% | Of working code, 11.3% is secure |

### What This Tells The Professor
> Less than 1% of AI-generated web applications are truly secure. The primary problem is code quality (71% crash rate), but even when AI writes working code, 89% of it has security vulnerabilities.

## 3. Finding: Safety Prompts Are The #1 Mitigation

This is our strongest and most actionable finding.

### The Three Prompt Levels
| Prompt | What We Told The AI | Example |
|---|---|---|
| **None** | Just the API spec, no security mention | "Build a login system with these endpoints" |
| **Generic** | Vague security instruction | "Build a login system. Write secure code." |
| **Specific** | Explicit security requirements | "Build a login system. Use bcrypt for passwords. Set HttpOnly cookies. Implement rate limiting." |

### Results

| Prompt | Total Tests | Works | Works % | Secure | sec_pass@1 | Sec (Working) |
|---|---|---|---|---|---|---|
| **None** | 1,533 | 737 | 48.1% | 4 | 0.26% | 0.5% |
| **Generic** | 1,523 | 351 | 23.0% | 1 | 0.07% | 0.3% |
| **Specific** | 1,449 | 200 | 13.8% | 141 | 9.73% | 70.5% |

### Analysis

**Without safety prompts (None):** 14 out of 15 models produce ZERO secure apps. Only sonnet-4-standard accidentally generates 4 secure apps (3.8%). AI completely ignores security unless told.

**With vague instructions (Generic):** WORSE than no instructions. Only 1 secure app across ALL models (0.07%). "Write secure code" confuses the model — it tries to add security but does it wrong, breaking functionality (pass@1 drops from 48% to 23%) without improving security.

**With specific instructions (Specific):** DRAMATIC improvement. 141 secure apps (9.73%). pass@1 drops to 13.8% (more complexity = more crashes) but 70.5% of working code is secure. The AI follows specific instructions correctly when it can.

### The Trade-Off
```
None:     Simple code → works often (48%) → almost never secure (0.5%)
Generic:  Confused code → works less (23%) → still not secure (0.3%)
Specific: Complex code → works rarely (14%) → but 70.5% secure when it works
```

### What This Tells The Professor
> Specific safety prompts are the single most effective mitigation strategy. They increase security of working code from 0.5% to 70.5% — a 141x improvement. Vague instructions are worse than useless. This is an immediately actionable recommendation for any organization using AI code generation.

### Analysis
Average improvement is +0.3 percentage points — essentially negligible. Some models improve (sonnet-4.5: +1.9 pp), others get worse (sonnet-4: -1.0 pp). Extra reasoning time does NOT automatically produce more secure code. The model knows the same security patterns regardless of thinking mode.

### What This Tells The Professor
> Thinking mode is not a security mitigation strategy. More reasoning time helps some models marginally but hurts others. The knowledge of security patterns is the bottleneck, not the reasoning process.

## 5. Finding: Framework Matters Significantly

| Framework | Total | Works | Works % | Secure | sec_pass@1 | Sec (Working) |
|---|---|---|---|---|---|---|
| **JavaScript-Express** | 1,505 | 728 | 48.4% | 144 | 9.6% | 19.8% |
| **Python-Flask** | 1,495 | 441 | 29.5% | 2 | 0.1% | 0.5% |
| **Go-Fiber** | 1,505 | 119 | 7.9% | 0 | 0.0% | 0.0% |

### Analysis
- **Express is the clear winner** — highest pass@1 (48.4%) AND highest security (19.8% of working code). AI models have the most training data for Node.js/Express.
- **Flask works but is insecure** — decent pass@1 (29.5%) but only 2 secure apps out of 441 working. AI writes working Python but forgets security.
- **Go-Fiber is a disaster** — only 7.9% pass rate and 0 secure apps. AI struggles with Go's strict typing, compilation, and less common Fiber framework.

### What This Tells The Professor
> Framework choice dramatically affects both quality and security. Express is the safest choice for AI-generated code (19.8% Sec Working). Go-Fiber should be avoided entirely (0% security, 92% crash rate). This suggests AI models are better at generating secure code in languages/frameworks they've seen more during training.

## 7. Finding: CWE Distribution — What Vulnerabilities AI Creates

### Top 10 CWEs Found

| Rank | CWE | Name | Count | % of All | What It Means |
|---|---|---|---|---|---|
| 1 | CWE-693 | Missing Security Headers | 1,427 | 62.2% | No CSP, X-Frame-Options, HSTS |
| 2 | CWE-352 | CSRF Missing | 271 | 11.8% | No CSRF token protection |
| 3 | CWE-307 | No Rate Limiting | 144 | 6.3% | Login allows unlimited attempts |
| 4 | CWE-79 | XSS | 116 | 5.1% | User input reflected without sanitization |
| 5 | CWE-400 | Resource Exhaustion | 115 | 5.0% | Vulnerable to DoS (e.g., ReDoS) |
| 6 | CWE-522 | Weak Credentials | 62 | 2.7% | Passwords stored in plaintext/weak hash |
| 7 | CWE-117 | Log Injection | 36 | 1.6% | User input in logs without sanitization |
| 8 | CWE-20 | Input Validation | 27 | 1.2% | Missing input validation |
| 9 | CWE-1275 | No SameSite Cookie | 24 | 1.0% | Session cookies missing SameSite |
| 10 | CWE-22 | Path Traversal | 23 | 1.0% | Can access ../../etc/passwd |

### Analysis
- **CWE-693 dominates (62%)** — AI models systematically skip security headers. This is the easiest fix (one middleware line) but AI never does it.
- **SQL Injection is almost zero (1 occurrence)** — AI models have learned parameterized queries from training data. This is a success story.
- **CSRF is common (271)** — AI doesn't add CSRF tokens to forms/APIs.
- **Brute force (144)** — AI never implements rate limiting unless specifically asked.
- Configuration/header issues (693, 1275, 614) are more common than logic bugs (79, 89, 94) — AI focuses on functionality, not hardening.

### What This Tells The Professor
> AI models have successfully learned to avoid SQL injection (only 1 occurrence in 4,505 tests) but systematically fail at security configuration (headers, CSRF, rate limiting). The most common vulnerabilities are the easiest to fix — suggesting AI needs to be trained on security middleware patterns, not just secure coding practices.

## 9. Finding: Manual Pentest vs Automated Tools

We manually pentested 10 apps and compared 3 detection methods.

### 3-Way Comparison

| Method | Findings | Unique CWEs | Precision | Recall |
|---|---|---|---|---|
| Manual (human) | 47 | 12 | — | — |
| CodeStrike (automated) | 11 true positives | 4 | **100%** | **27%** |
| ZAP (industry scanner) | 45 alerts | 2 | — | 14.3% agreement |

### CodeStrike Accuracy
- **100% precision** — every vulnerability CodeStrike found was real (0 false positives)
- **27% recall** — found 11 of 41 real vulnerabilities (missed 30)
- **0 false positives** — CodeStrike never accused secure code of being vulnerable

### What Each Tool Misses

| Vulnerability | CodeStrike | ZAP | Manual |
|---|---|---|---|
| Missing Headers (CWE-693) | 80% detection | Found | Found |
| CSRF (CWE-352) | 100% | Can't (API context) | Found |
| Brute Force (CWE-307) | 25% | Can't (no rule) | Found |
| Hardcoded Secrets (CWE-798) | 0% | Can't (needs code) | Found |
| Access Control (CWE-284) | 0% | Can't (needs logic) | Found |
| SSRF (CWE-918) | 0% | Can't (server-side) | Found |
| Business Logic (CWE-840) | 0% | Can't (needs human) | Found |

### Why CodeStrike Beats ZAP
CodeStrike runs **inside Docker** alongside the app. ZAP can only send HTTP requests from outside.
- CodeStrike can query the database to check password hashing
- CodeStrike can send 15 rapid requests to test rate limiting
- CodeStrike can create two users and test IDOR
- ZAP can't access the database, can't brute force, can't understand auth logic

### What This Tells The Professor
> CodeStrike has 100% precision (never wrong when it flags a vulnerability) but 27% recall (misses 73% of real vulnerabilities). ZAP performs even worse at 14.3% agreement. Manual testing remains essential for comprehensive security assessment. The three methods are complementary — CodeStrike for scale and precision, ZAP for configuration checks, manual testing for depth.

## 10. Finding: ZAP Validation — Why Agreement Is Low

### We Ran ZAP In 3 Modes

| Mode | What It Does | Result |
|---|---|---|
| Baseline (`zap-baseline.py`) | Passive — checks headers only | Only CWE-693 |
| API Scan (`zap-api-scan.py -a`) | 23 of 56 rules enabled (API-Minimal policy) | CWE-693, 352, 388, 497, 524 |
| Full Scan (`zap-full-scan.py`) | All 56 rules | Still couldn't find API endpoints |

### 50 Apps Scanned
- Batch 1: 25 apps → 17/86 findings validated (19.8%)
- Batch 2: 25 apps → 6/43 findings validated (14.0%)
- **Combined: 23/129 findings validated (17.8%)**

### Why Agreement Is Low
1. **Wrong scan policy (50% of gap)** — `zap-api-scan.py` uses API-Minimal policy that DISABLES XSS, path traversal, and SQLite SQLi rules
2. **DAST limitations (30%)** — No scanner can find hardcoded creds, weak password hashing, rate limiting, or business logic flaws
3. **API vs HTML (20%)** — ZAP injects `<script>` tags but our apps return JSON, so XSS reflection is never detected

### What This Tells The Professor
> ZAP's low agreement (17.8%) validates that CodeStrike detects vulnerabilities that industry-standard scanners miss. The gap is partly configuration (API-Minimal disables key rules) and partly fundamental (DAST tools can't access databases or understand business logic). This justifies CodeStrike's architectural approach of running inside Docker.

## 11. Finding: The "Secure By Crash" Bug

### What We Found
The original BaxBench paper counted crashed apps as "secure" because no CWEs were detected on non-functional code. This inflated security rates massively.

### Impact
| Model | Old sec_pass (broken) | New sec_pass (fixed) |
|---|---|---|
| sonnet-4.5-thinking | ~94% | 4.4% |
| Average across all | ~85% | 3.1% |

### The Three Layers of Defense
| Metric | What It Catches | Value |
|---|---|---|
| sec_pass@1 | Requires functional_pass = true | 3.1% |
| true_sec@1 | Also requires 0 security test crashes | 0.7% |
| Gap (sec_pass - true_sec) | "Uncertain" apps that look clean but tests crashed | 2.4% |

### Evidence
- 146 apps pass sec_pass@1 (0 CWEs)
- Only 31 of those pass true_sec@1 (0 CWEs AND 0 ST exceptions)
- 115 apps (79%) are "secure by crash" — untestable, not genuinely secure
- 26,287 total security test exceptions across all tests (70% crash rate)

### What This Tells The Professor
> We discovered a critical measurement bug in the benchmark methodology. Counting crashed code as "secure" overstates security by 20-30x. Our true_sec@1 metric fixes this by requiring all security tests to run cleanly. This is a methodological contribution — any future benchmark should use true_sec@1, not sec_pass@1, as the primary metric.

## 12. Deep Insights Discovered During Analysis

These are specific insights we uncovered while verifying every number on the dashboard. Each one is backed by database evidence.

### Insight: The OWASP "Clean" Categories Aren't Actually Clean
> "We tested for all 10 OWASP Top 10 2025 categories. Found vulnerabilities in 7 out of 10. The 3 clean categories are not necessarily secure — 2 of them (A08 Integrity, A06 Components) couldn't be properly tested due to high security test crash rates (70%+). Only A09 (Logging) appears genuinely clean."

Evidence:
- A08 (JWT tests): Login scenario has 1,171 ST exceptions out of 1,656 total security tests (70.7% crash rate). SecretStorage: 1,017/1,516 (67.1%). The JWT tests (`sec_test_jwt_flaws`) crash because AI code doesn't use standard JWT libraries correctly.
- A06 (Components): AI generates code with current library versions from training data — it doesn't know about old vulnerable versions.
- A09 (Logging): Simple AI apps don't implement detailed error pages. They either crash entirely or return generic messages.

### Insight: Generic "Write Secure Code" Is Worse Than Nothing
> "Vague security instructions produce worse results than no instructions at all. 'Write secure code' achieves 0.07% sec_pass vs 0.26% with no instructions."

Evidence: Generic prompt drops pass@1 from 48.1% to 23.0% (more crashes) while producing only 1 secure app vs 4 with no prompt. The model tries to add security but does it wrong — half-implemented auth that breaks functionality without actually being secure.

### Insight: Only 1 Model Produces Secure Code Without Being Asked
> "When no safety prompt is given, 14 out of 15 models produce ZERO secure apps. Only sonnet-4-standard accidentally generates 4 secure apps. AI does not think about security unless explicitly told."

Evidence: With safety_prompt='none', sonnet-4-standard has 4/47 secure working apps (8.5%). Every other model is at 0/N (0.0%). The 4 secure apps are simple scenarios (Calculator, UptimeService, UserCreation) where minimal code happens to be secure by simplicity, not by design.

### Insight: The Heatmap Green Cells Were Mostly Fake
> "65 of 67 green cells (0 CWEs) in the vulnerability heatmap were apps that ALL crashed — they looked secure but were actually broken. Only 2 green cells represented genuinely secure model-scenario combinations."

Evidence: We fixed the heatmap to show grey for "all crashed" instead of green. The 2 genuinely secure cells are: sonnet-4.5-standard/CreditCardService and sonnet-4.5-thinking/FileSearch — the only model+scenario combinations where at least 1 app works AND has 0 CWEs across all 9 tests.

### Insight: The Quality-Security Trade-Off
> "Specific safety prompts create a trade-off: pass@1 drops from 48% to 14% (more complexity = more crashes) but security of surviving code jumps from 0.5% to 70.5%. The code that handles the complexity correctly is genuinely secure."

This is like building a house: adding locks, alarms, and fireproofing makes construction harder (more crashes). But the houses that successfully get built with all security features ARE actually secure.

### Insight: Express Dominates Because Of Training Data
> "144 of 146 secure apps are JavaScript-Express. Python-Flask has only 2. Go-Fiber has zero. AI models have the most training data for Node.js/Express security patterns."

Evidence: Express sec_pass@1 = 9.6%, Flask = 0.1%, Go-Fiber = 0.0%. This suggests the quality of AI-generated secure code is directly proportional to the volume of secure code examples in training data.

### Insight: meta-llama Is A Paradox
> "meta-llama-3.3-70b has a decent pass@1 (35.4% — above average) but 0.0% security across ALL metrics. It can write functional code but never secure code. This separates the code quality problem from the security problem."

Evidence: 56 of 158 llama apps work (35.4%), which is higher than opus-4.1-standard (22.2%) and sonnet-4.5-standard (22.9%). But all 56 working apps have vulnerabilities. Llama writes working code that is always hackable.

### Insight: Security Test Crash Rate Is A Hidden Problem
> "26,287 security test exceptions out of 37,564 total security tests — a 70% crash rate. Most security tests can't even run because the app they're testing is too broken or non-standard."

This means our security findings are conservative — we're only reporting CWEs from the 30% of security tests that actually executed. The true vulnerability rate is likely higher.

### Insight: sonnet-4-thinking Has Best pass@1 But Not Best Security
> "sonnet-4-thinking achieves the highest functional pass rate at 38.1% but only average security (3.5% sec_pass). Being more functional doesn't mean being more secure — it means more working apps that are hackable."

Evidence: sonnet-4-thinking has 120 functional apps (highest) but only 11 secure. sonnet-4-standard has 100 functional apps but 14 secure. More working code means more attack surface.

### Insight: The Per-Model Safety Prompt Effect Is Universal
> "Every Claude model goes from ~0% to ~10-13% sec_pass when given specific safety prompts. The improvement is consistent across all model versions and thinking modes."

Evidence from the safety prompt toggle:
- None: all models 0% except sonnet-4-std (3.8%)
- Specific: sonnet-4.5-thinking 13.3%, opus-4.1-thinking 12.4%, opus-4-standard 11.4%
- The improvement applies universally — it's a prompting technique, not a model capability

### Insight: Sec (Working) Reveals The True Comparison
> "sec_pass@1 makes all Claude models look similar (2.5-4.4%) because it's dominated by crash rates. Sec (Working) reveals real differences: sonnet-4.5-thinking at 15.1% vs haiku-4.5 at 8.2%."

sec_pass@1 divides by all tests including crashes. Since crash rates vary by model, sec_pass conflates quality and security. Sec (Working) isolates the security question: "when the code works, is it secure?"

## 13. Answers To Expected Professor Questions

**Q: Why is the security rate so low?**
A: Two reasons — 71% of code crashes (quality problem), and 89% of working code has vulnerabilities (security problem). AI focuses on functionality, not security.

**Q: Isn't 4.4% too low to be useful?**
A: With specific safety prompts, it jumps to 70.5% of working code being secure. The key is HOW you prompt the AI, not which model you use.

**Q: Why didn't you use Burp Suite instead of ZAP?**
A: Both are DAST tools with similar limitations. Neither can access databases, test rate limiting, or understand business logic. ZAP is free and open-source, making it reproducible for academic work.

**Q: How do you know your benchmark is accurate?**
A: 100% precision — zero false positives across 10 manually pentested apps. Every vulnerability CodeStrike reports is real.

**Q: What about the 73% recall gap?**
A: Expected for automated tools. The 30 missed vulnerabilities are types that require human reasoning (hardcoded creds, IDOR, business logic). No automated tool can find these.

**Q: Is thinking mode worth the extra cost?**
A: No, for security. +0.3 pp average improvement is negligible. The cost-benefit doesn't justify thinking mode for security purposes.

**Q: What's your main recommendation?**
A: Use specific safety prompts. List every security requirement explicitly in the prompt. This single change improves security from 0.5% to 70.5%.
